Securing your endpoint

Risks & how to lock it down

The parsing endpoint runs on a public URL and calls OpenAI with your key. Anyone who learns the URL could otherwise run up your bill. Reduce that risk:

  • Restrict the origin. Set ALLOWED_ORIGIN on your server to your app's address. Cross-site browser calls are then rejected.
  • Add a shared secret. Set IMPORT_SHARED_SECRET on your server and the same value in the field above. Requests without it (including curl) get a 403.
  • Cap spending. Set a hard monthly usage limit in your OpenAI account — the reliable backstop, since any public endpoint can be probed.
  • Keep the URL private. Don't post your endpoint publicly; treat it like a key.
  • What's sent: only the statement PDF and your categories — to your server, then OpenAI. It isn't logged or stored by the endpoint, and your encrypted budget is never sent.

How your security works

Your data, and how it's protected

A plain-language summary so you can make good choices.

  • Encrypted on this device. Your budget is stored using AES-256-GCM. The key is stretched from your password with PBKDF2-SHA256 (1,000,000 rounds). Nothing is sent to any server in normal use.
  • Your password is everything. There is no recovery and no backdoor. If someone copies the encrypted file from your device, the only thing standing between them and your data is how hard your password is to guess.
  • The real threat is offline guessing. An attacker with the file can try millions of common passwords on their own hardware. A word plus a year (like “Football2019”) can fall in seconds. That class of password is now blocked at setup.
  • Pick 4 random words. Something like otter lantern copper meadow is easy to remember and would take centuries to crack. Avoid names, dates, and anything reused from another account.
  • The PIN is only a screen lock. It hides the open screen on this device. It does not add encryption — your password is still what protects the data when the app is reopened.
  • Want it un-crackable even if the file is stolen? Turn on online high-security mode below. Your password alone can no longer unlock the file — your own server must take part in every attempt.

Online high-security mode

Optional. Makes the encrypted file impossible to brute-force offline by requiring your own server to take part in unlocking.

How it works. Your key becomes HMAC(server secret, your password hash). The server never sees your password or your data and can't decrypt anything — but because its secret is needed for every guess, a stolen file alone is useless.
  • Trade-off: unlocking needs your server reachable. After you unlock, normal edits still save offline.
  • No recovery: if the server secret (KDF_PEPPER) is lost or changed, the data can't be decrypted. Keep a plain backup somewhere safe first.
  • Setup: deploy /api/pepper and set a long random KDF_PEPPER on your server (see the README).
